About Project

With the development of technology, people's need for applications that will make their daily lives easier.

Due to the growing number of tourists, and the increase in the population in Porto, the number of public transport users has also increased.

A large number of Porto residents use city buses for transportation. At the moment, there is no possibility to follow the current location of the bus, nor to view its route within an application.

Therefore, we decided to create applications that will facilitate the use of city buses in Porto, so that users can make smarter use of the time they may have to spend at bus stops. In addition to viewing the current location and route of the bus, the user will be able to set an alarm that will sound when the bus is at a certain distance from the current location.

Cenarios

Before we started developing the system, we set three basic scenarios that will be solved with the help of our system:


Finding route of bus

In this scenario, the user will select the bus for which he wants to check the route. The system will display the route using google maps, with each station marked with a marker

Tracking bus

In this scenario, the user will select the bus for which he wants to check the current location. The system will display the map using google maps, where the current location of the bus will be marked with an icon, and in addition, the entire bus route will be visible.

Setting alarm

This scenario involves using multiple microservices to ultimately get an alarm. Namely, the user selects the bus for which he wants to receive a notification when he is at a certain distance from his current location. When the bus reaches the set distance from the current location, the user will receive a popup that will inform him about it.

Arquitecture



Rest API simulator (data simulator) is a python script that reads data collected from sensors stored into a csv file. Every row of a csv file is readed every 1.5 second and sended to Kafka topic ESP13_bus_data.

Realtime service is Spring boot application that is credited with communicating with the frontend. Realtime service contains Kafka consumer who receives messages on topic ESP13_bus_data, and if bus id of the obtained data is equal to id of desired bus, getted from frontend, then service sends the new obtained latitude and longitude. Other purpose of Realtime service is that its sending alarm to frontend consumed in on a kafka topic "ESP13_distance_alarm." Connection between frontend and realtime time service is obtained by using Server Sent Event.

Distance service is a Spring boot application which the main objective is to set an alarm for a particular bus. Kafka consumer made within the service get messages on topic ESP13_bus_data. After receiving the data, it calculates (if the data whose bus id matches the id for which the alarm is set) the distance between the obtained location and the current location of the user. If the location is less than 1000 meters, an alarm is produced to Kafka topic ESP13_distance_alarm.

Database service is Spring boot application whose purpose is to consume Kafka messages with the topic ESP13_bus_data and save it into the database. In addition, this application contains routes from which it is possible to get all bus ids, their routes and history data. Database is a mysql database that contains historical data about positions of all busses. The database is filled up after each Kafka message is consumed on topic ESP13_bus_data.

Frontend is React application whose purpose is to present information about buses. First screen contains table with all buses ids. For each bus it is possible to define alarm, track his current position or see bus route. When user wants to track some bus current position or just see his route, application will open modal that will contain google map, on which with the help of react-google-map package and it functions will be presented routes, stop points and current location of bus.

@Given −

Describes the pre-requisite for the test to be executed.


@When −

Defines the trigger point for the test scenario execution.


@Then −

Expected result


Knowing this, we start doing our tests. For the database we use the tests to see if we were getting the expected data, for example, to see if we got proper values and not empty ones, to see if the services were being started and to see if a value was saved into the database after being inserted. To use the database, we opted to use Mysql, as it was required to use an in-memory database. For this we added the dependency “mysql-connector-java. Although the tests are not as critical as in a full blown project, it is very important to take advantage of them to see early mistakes, as well for future project scalability.
